# LLVM IR and Bitcode Output Guide

The CURSED compiler provides comprehensive support for generating LLVM Intermediate Representation (IR) and bitcode files from CURSED programs. This functionality enables debugging, inspection, alternative compilation workflows, and binary distribution.

## Overview

The IR output system supports:
- **Human-readable LLVM IR (.ll files)** - Perfect for debugging and learning
- **LLVM bitcode (.bc files)** - Efficient binary format for distribution and linking
- **Both formats simultaneously** - Maximum flexibility
- **Single-module compilation** - Standard programs
- **Separate compilation** - Multi-package projects with modular output

## Basic Usage

### Command Line Interface

Generate LLVM IR from a CURSED program:
```bash
cursed --emit-ir program.csd
```

Generate LLVM bitcode:
```bash
cursed --emit-bc program.csd
```

Generate both formats:
```bash
cursed --emit-both program.csd
```

### Advanced Options

Specify custom output directory:
```bash
cursed --emit-ir --output-dir build program.csd
```

Custom base name for output files:
```bash
cursed --emit-ir --base-name my_program program.csd
```

Optimize before output:
```bash
cursed --emit-ir --optimize program.csd
```

Don't preserve directory structure:
```bash
cursed --emit-ir --no-preserve-structure src/examples/program.csd
```

Disable debug comments in IR:
```bash
cursed --emit-ir --no-debug-comments program.csd
```

## Output Formats

### LLVM IR (.ll files)

Human-readable text format that shows the LLVM intermediate representation:

```llvm
; Generated by CURSED compiler
; LLVM IR output for debugging and inspection

define i32 @main() {
entry:
  %call = call i32 @add(i32 5, i32 3)
  %call1 = call void @print_int(i32 %call)
  ret i32 0
}

define i32 @add(i32 %a, i32 %b) {
entry:
  %add = add i32 %a, %b
  ret i32 %add
}
```

Benefits:
- Easy to read and understand
- Perfect for debugging compilation issues
- Great for learning LLVM IR structure
- Can be modified and recompiled with LLVM tools

### LLVM Bitcode (.bc files)

Binary format that's compact and efficient:

Benefits:
- Smaller file size than text IR
- Faster to parse and process
- Perfect for distribution and linking
- Can be optimized and compiled with LLVM tools
- Platform-independent intermediate format

## Configuration Options

### IrOutputConfig Structure

```rust
pub struct IrOutputConfig {
    /// Output format (IR, bitcode, or both)
    pub format: IrOutputFormat,
    /// Output directory for generated files
    pub output_dir: PathBuf,
    /// Whether to preserve directory structure from input
    pub preserve_structure: bool,
    /// Whether to optimize the module before output
    pub optimize: bool,
    /// Base name for output files (when not preserving structure)
    pub base_name: Option<String>,
    /// Whether to include debug comments in IR output
    pub include_debug_comments: bool,
}
```

### Output Formats

- `IrOutputFormat::LlvmIr` - Generate only .ll files
- `IrOutputFormat::Bitcode` - Generate only .bc files  
- `IrOutputFormat::Both` - Generate both .ll and .bc files

## Directory Structure

### Preserve Structure (Default)

When `preserve_structure` is true, the output mirrors the input structure:

```
Input:
  src/
    examples/
      hello.csd
    utils/
      math.csd

Output:
  output/
    src/
      examples/
        hello.ll
        hello.bc
      utils/
        math.ll
        math.bc
```

### Flat Structure

When `preserve_structure` is false, all files go to the output directory:

```
Output:
  output/
    hello.ll
    hello.bc
    math.ll
    math.bc
```

## Separate Compilation Support

For multi-package projects, the IR output system provides additional capabilities:

### Per-Package Output

Generate IR/bitcode for each package individually:

```rust
use cursed::codegen::llvm::{SeparateIrOutput, SeparateIrOutputConfig};

let config = SeparateIrOutputConfig {
    per_package: true,
    linked_output: false,
    package_subdir: "packages".to_string(),
    ..Default::default()
};
```

Output structure:
```
output/
  packages/
    package1/
      package1.ll
      package1.bc
    package2/
      package2.ll
      package2.bc
```

### Linked Output

Generate IR/bitcode for the final linked program:

```rust
let config = SeparateIrOutputConfig {
    per_package: true,
    linked_output: true,
    linked_subdir: "linked".to_string(),
    ..Default::default()
};
```

Output structure:
```
output/
  packages/
    package1/
      package1.ll
      package1.bc
    package2/
      package2.ll
      package2.bc
  linked/
    program.ll
    program.bc
```

## Programming Interface

### Basic IR Generation

```rust
use cursed::codegen::llvm::{IrOutputGenerator, IrOutputConfig, IrOutputFormat};
use inkwell::context::Context;

let context = Context::create();
let config = IrOutputConfig {
    format: IrOutputFormat::Both,
    output_dir: PathBuf::from("output"),
    ..Default::default()
};

let generator = IrOutputGenerator::new(&context, config);
let generated_files = generator.generate_from_program(&program, "input.csd")?;

println!("Generated {} files", generated_files.all_files().len());
```

### Convenience Functions

```rust
use cursed::codegen::llvm::{generate_ir_output_default, generate_ir_output};

// With default configuration
let files = generate_ir_output_default(&context, &program, "input.csd")?;

// With custom configuration  
let files = generate_ir_output(&context, &program, "input.csd", config)?;
```

### Separate Compilation

```rust
use cursed::codegen::llvm::{generate_separate_ir_output, SeparateIrOutputConfig};

let generated = generate_separate_ir_output(
    &context,
    &compiler,
    &modules,
    Some(&linked_module),
    config,
    "my_program"
)?;

generated.print_summary();
```

## Use Cases

### Debugging Compilation Issues

When your CURSED program doesn't compile or behave as expected:

1. Generate LLVM IR: `cursed --emit-ir --include-debug-comments program.csd`
2. Examine the generated IR to understand what the compiler produced
3. Look for unexpected function signatures, missing functions, or incorrect logic
4. Compare with expected LLVM IR patterns

### Learning LLVM IR

Study how CURSED language constructs translate to LLVM IR:

```bash
# Generate IR for a simple program
cursed --emit-ir examples/ir_output_demo.csd

# Examine the generated .ll file
cat output/ir_output_demo.ll
```

### Alternative Compilation Workflows

Use external LLVM tools with generated IR/bitcode:

```bash
# Generate bitcode
cursed --emit-bc program.csd

# Optimize with LLVM opt
opt -O3 output/program.bc -o optimized.bc

# Compile to assembly with LLC
llc optimized.bc -o program.s

# Link with external libraries
clang program.s -lcursed_runtime -o program
```

### Binary Distribution

Distribute pre-compiled modules as bitcode:

```bash
# Compile library to bitcode
cursed --emit-bc --output-dir dist lib/math.csd

# Users can link the bitcode into their programs
llvm-link user_program.bc dist/math.bc -o combined.bc
```

### Performance Analysis

Generate optimized IR to analyze performance:

```bash
cursed --emit-ir --optimize --output-dir analysis program.csd
# Analyze the optimized IR for performance bottlenecks
```

## Integration with Build Systems

### Makefile Integration

```makefile
CURSED_SOURCES = $(wildcard src/*.csd)
IR_FILES = $(CURSED_SOURCES:src/%.csd=output/%.ll)
BC_FILES = $(CURSED_SOURCES:src/%.csd=output/%.bc)

.PHONY: ir bitcode

ir: $(IR_FILES)
bitcode: $(BC_FILES)

output/%.ll: src/%.csd
	cursed --emit-ir --output-dir output $<

output/%.bc: src/%.csd
	cursed --emit-bc --output-dir output $<
```

### Custom Build Scripts

```bash
#!/bin/bash
# build_ir.sh - Generate IR for all CURSED files

OUTPUT_DIR="build/ir"
OPTIMIZE=false

for file in src/**/*.csd; do
    if [ "$OPTIMIZE" = true ]; then
        cursed --emit-both --optimize --output-dir "$OUTPUT_DIR" "$file"
    else
        cursed --emit-both --output-dir "$OUTPUT_DIR" "$file"
    fi
done

echo "Generated IR and bitcode in $OUTPUT_DIR"
```

## Tips and Best Practices

### File Organization

- Use `--preserve-structure` for projects with complex directory layouts
- Use flat output structure for simple projects or when integrating with tools that expect flat directories
- Organize separate compilation output with meaningful subdirectory names

### Optimization

- Use `--optimize` when generating IR for performance analysis
- Generate both optimized and unoptimized IR to compare compiler transformations
- Remember that optimization can make IR harder to read for debugging

### Debug Information

- Keep `--include-debug-comments` enabled when debugging
- Use descriptive `--base-name` options for multiple output files
- Include source file paths in your build documentation

### Error Handling

- Check that output directories are writable before starting large compilations
- Verify generated files exist and have expected content
- Use the programming interface for better error reporting in automated tools

## Troubleshooting

### Common Issues

**Output directory not found:**
```bash
Error: Failed to create output directory: Permission denied
```
Solution: Ensure the output directory is writable or specify a different location.

**No files generated:**
```bash
Error: No output files generated
```
Solution: Check that the input CURSED program compiles successfully before generating IR.

**Large IR files:**
```
Warning: Generated IR file is very large (>10MB)
```
Solution: Consider using bitcode format or optimizing the CURSED program structure.

### Debugging IR Output

1. Start with simple CURSED programs to verify IR generation works
2. Compare generated IR with expected LLVM patterns
3. Use LLVM tools to validate generated IR: `llvm-as < output.ll`
4. Check LLVM version compatibility if IR looks unusual

## Example Output

For the example program `examples/ir_output_demo.csd`, the compiler generates:

**ir_output_demo.ll (excerpt):**
```llvm
; Generated by CURSED compiler
; LLVM IR output for debugging and inspection

@GlobalCounter = global i32 0

define i32 @increment(i32 %amount) {
entry:
  %0 = load i32, i32* @GlobalCounter
  %1 = add i32 %0, %amount
  store i32 %1, i32* @GlobalCounter
  ret i32 %1
}

define i32 @fibonacci(i32 %n) {
entry:
  %0 = icmp sle i32 %n, 1
  br i1 %0, label %if.then, label %if.else

if.then:
  ret i32 %n

if.else:
  %1 = sub i32 %n, 1
  %2 = call i32 @fibonacci(i32 %1)
  %3 = sub i32 %n, 2
  %4 = call i32 @fibonacci(i32 %3)
  %5 = add i32 %2, %4
  ret i32 %5
}
```

This IR shows:
- Global variable declarations
- Function definitions with proper LLVM types
- Control flow with basic blocks and branches
- Recursive function calls
- Memory operations (load/store)

The corresponding bitcode file `ir_output_demo.bc` contains the same information in binary format, suitable for further processing with LLVM tools.
