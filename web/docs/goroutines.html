Goroutines

Goroutines in CURSED are the moment! They're lightweight threads that let your code multitask with minimal drama. It's giving concurrent execution vibes without the complexity.

Goroutine Characteristics

  • ⚡ Lightweight: Goroutines use minimal memory (a few KB vs MB for OS threads)
  • 📈 Scalable: Programs can run thousands or millions of goroutines
  • 🔄 Multiplexed: Many goroutines share fewer OS threads
  • 🔍 Auto-managed: The runtime handles scheduling and context switching
  • ✨ Simple syntax: Just add stan before a function call

Capture Loop Variables Correctly

One of the classic goroutine gotchas is the loop variable capture. It's not giving the vibes you expect:

fr fr WRONG WAY: will likely print the same number multiple times
bestie i := 0; i < 5; i++ {
    stan slay() {
        vibez.spill(i)  fr fr Captures reference to i, not its value
    }()
}

fr fr RIGHT WAY: capture the current value
bestie i := 0; i < 5; i++ {
    val := i  fr fr Create a new variable for each iteration
    stan slay() {
        vibez.spill(val)  fr fr Correct value for each goroutine
    }()
}

fr fr ALTERNATIVE: pass as parameter to the function
bestie i := 0; i < 5; i++ {
    stan slay(val normie) {
        vibez.spill(val)  fr fr Parameter gets the value at call time
    }(i)
}

Sharing Data Between Goroutines

There are several ways to share data between goroutines, each with different vibes:

Channels

The main way to communicate - gives synchronized messaging vibes

For shared memory access - gives exclusive access energy

Goroutine Best Practices

  • Always ensure proper synchronization (WaitGroup, channels, etc.)
  • Be cautious with shared variables - use channels or mutual exclusion
  • Consider limiting the number of goroutines for CPU-bound work
  • Use buffered channels when you know the exact number of messages
  • Watch for goroutine leaks (goroutines that never terminate)
  • Use select with timeouts for operations that might block indefinitely
  • Consider using worker pools for controlled parallelism
Cancellation Pattern

        mood <-done:
