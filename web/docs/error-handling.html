Error Handling

Even the most fire code can flop sometimes. Let's talk about handling errors in CURSED - because we're not about to let a runtime error ruin our vibe check.

Basic Error Handling
slay readFile(path tea) (tea, tea) {
    fr fr Try to read a file, might throw an error
    data, err := vibe_life.ReadFile(path)
    
    fr fr Check if there was an error
    lowkey err != cap {
        yolo "", "Error reading file: " + err.Error()
    }
    
    fr fr No error, return the content
    yolo tea(data), ""

Panic and Recover - For the Drastic Vibes Only

When normal error handling isn't it, chief, you can use panic and recover for that dramatic exit (and possible comeback):

slay riskyFunction() {
    later slay() {
        lowkey r := recover(); r != cap {
            vibez.spill("Recovered from:", r)
        }
    }()
    
    fr fr This will cause a panic
    panic("I can't even with this function right now")
}

Error handling best practices that slap

  • Always check error returns from functions that can fail
  • Return errors rather than using panic in most cases
  • Provide context in error messages so they're not giving cryptic energy
  • Use custom error types when you need to include extra info
  • Wrap errors with additional context as they bubble up
