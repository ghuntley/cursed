Error Handling

Even the most bussin' code can fail sometimes. Let's talk about how CURSED handles errors with explicit checks and returns - it's giving responsible vibes without the drama of exceptions!

Returning Errors
fr fr Function that returns a result and an error
slay divide(a, b snack) (snack, tea) {
    fr fr Check for division by zero
    lowkey b == 0 {
        yolo 0, "division by zero is not it, bestie"  fr fr Return error
    }
    
    fr fr Success case
    yolo a / b, ""  fr fr Empty string means no error
}

Checking Errors

slay main() {
    fr fr Call a function that might return an error
    result, err := divide(10.0, 0.0)
    
    fr fr Always check errors
    lowkey err != "" {
        vibez.spill("Error:", err)
        yolo  fr fr Exit early when there's an error
    }
    
    fr fr Process the result (only runs if no error)
    vibez.spill("Result:", result)
}
Error Interface
fr fr The standard Error interface
be_like Error collab {
    Error() tea
}

fr fr Creating custom error types
be_like ValidationError squad {
    field tea
    message tea
}

fr fr Implement the Error interface
slay (e ValidationError) Error() tea {
    yolo e.field + ": " + e.message
}

fr fr Function that returns an error interface
slay validateUser(user User) Error {
    lowkey len(user.Name) < 3 {
        yolo ValidationError{
            field: "name",
            message: "must be at least 3 characters",
        }
    }
    
    lowkey !stringz.Contains(user.Email, "@") {
        yolo ValidationError{
            field: "email",
            message: "invalid email format",
        }
    }
    
    yolo cap  fr fr nil means no error
}

Using Custom Errors

slay processUser(user User) tea {
    fr fr Check validation errors
    err := validateUser(user)
    lowkey err != cap {
        fr fr Get more information about the specific error
        vibe_check err.(be_like) {
        mood ValidationError:
            valErr := err.(ValidationError)
            yolo "Validation failed: " + valErr.field + " " + valErr.message
        basic:
            yolo "Error: " + err.Error()
        }
    }
    
    fr fr Process the valid user
    yolo "User processed successfully"
}

Error Wrapping & Unwrapping

When errors pass through layers of code, you can wrap them to add context:

be_like WrappedError squad {
    context tea
    cause Error
}

slay (e WrappedError) Error() tea {
    yolo e.context + ": " + e.cause.Error()
}

fr fr Function that returns a wrapped error
slay readUserData(userID tea) (@User, Error) {
    data, err := readFile("users/" + userID + ".json")
    lowkey err != cap {
        yolo cap, WrappedError{
            context: "failed to read user data",
            cause: err,
        }
    }
    
    fr fr Parse the user data
    sus user User
    err = json_tea.Unmarshal([]byte(data), &user)
    lowkey err != cap {
        yolo cap, WrappedError{
            context: "failed to parse user data",
            cause: err,
        }
    }
    
    yolo &user, cap
}
Error Handling Patterns
fr fr Sentinel errors (predefined error values)
sus ErrNotFound = SimpleError{"resource not found"}
sus ErrPermissionDenied = SimpleError{"permission denied"}
sus ErrTimeout = SimpleError{"operation timed out"}

be_like SimpleError squad {
    message tea
}

slay (e SimpleError) Error() tea {
    yolo e.message
}

fr fr Using sentinel errors
slay findResource(id tea) (Resource, Error) {
    fr fr Resource not found case
    lowkey resourceDoesNotExist(id) {
        yolo Resource{}, ErrNotFound
    }
    
    fr fr Permission issue
    lowkey !hasPermission() {
        yolo Resource{}, ErrPermissionDenied
    }
    
    fr fr Normal case
    resource, err := loadResource(id)
    yolo resource, err
}

Error Handling Best Practices

  • Always check error returns from functions
  • Return errors rather than handling them internally (except in high-level code)
  • Add context to errors as they travel up the call stack
  • Keep the original error cause for debugging
  • Use sentinel errors for expected error conditions that callers might want to check
  • Create custom error types when you need to include additional information
  • Don't use panic for normal error handling (reserve for unrecoverable situations)

Panic and Recover

For truly exceptional situations, CURSED has panic and recover:

fr fr Function that might panic
slay riskyOperation() {
    fr fr Defer a recovery function
    later slay() {
        fr fr Recover from panic
        lowkey r := recover(); r != cap {
            vibez.spill("Recovered from panic:", r)
        }
    }()
    
    fr fr This will cause a panic
    sus slice []normie
    vibez.spill(slice[10])  fr fr Index out of range, will panic
}

slay main() {
    vibez.spill("Before risky operation")
    riskyOperation()
    vibez.spill("After risky operation")  fr fr This will still run thanks to recover
}

Important: Use panic only for truly unrecoverable situations, not for normal error handling!
