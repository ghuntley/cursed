Packages & Imports

Let's talk about organizing your code with packages - it's the modular moment! In CURSED, we don't just import, we yeet those dependencies into our code.

Package Declaration
fr fr Every source file must declare its package
vibe main

fr fr Library files often use directory name as package name
vibe stringutils

fr fr Subpackages follow directory structure
vibe vibez/format

Importing Packages

fr fr Single import
yeet "vibez"

fr fr Multiple imports
yeet (
    "vibez"
    "vibe_life"
    "timez"
    "mathz"
)

fr fr Import with alias
yeet str "stringz"

fr fr Dot import (not recommended)
yeet . "vibez"  fr fr Now you can use spill() instead of vibez.spill()
Package Structure Example
fr fr In file myapp/user/user.csd
vibe user

fr fr Exported type (capital first letter)
be_like User squad {
    Username tea
    Email tea
    password tea  fr fr Unexported field (lowercase)
}

fr fr Exported function
slay NewUser(username tea, email tea, password tea) @User {
    yolo &User{
        Username: username,
        Email: email,
        password: password,
    }
}

fr fr Unexported function
slay validateEmail(email tea) lit {
    fr fr Implementation details
    yolo based
}
Using the Package
fr fr In file myapp/main.csd
vibe main

yeet (
    "vibez"
    "myapp/user"  fr fr Import our custom package
)

slay main() {
    fr fr Create a new user using the exported function
    alice := user.NewUser("alice", "alice@example.com", "securepass")
    
    fr fr Access exported fields
    vibez.spillf("User: %s, Email: %s", alice.Username, alice.Email)
    
    fr fr Cannot access unexported fields
    fr fr This would cause a compilation error:
    fr fr vibez.spill(alice.password)
    
    fr fr Cannot access unexported functions
    fr fr This would cause a compilation error:
    fr fr user.validateEmail("test@example.com")
}

Package Visibility Rules

The tea on what's public and what's private in CURSED:

  • Exported (Public): Identifiers that start with an uppercase letter
  • Unexported (Private): Identifiers that start with a lowercase letter
  • Exported entities can be accessed from other packages
  • Unexported entities can only be accessed within the same package

Package Organization That Slays

Some package structure best practices that are giving main character energy:

  • Keep packages small and focused on a single responsibility
  • Name packages using short, lowercase, single-word names
  • Avoid package names that are too generic (e.g., "utils")
  • Organize by functionality, not by file type
  • Consider creating a pkg/ directory for reusable libraries
  • Use internal/ directories for packages that shouldn't be imported by other projects
