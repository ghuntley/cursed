Generics in CURSED

Generics let you write code that works with any type. It's like having a bestie who gets along with everyone in your friend group.

Why Generics Are The Main Character

  • ♻️ Code Reusability: Write a function or data structure once, use it with any type
  • 🛡️ Type Safety: Get compile-time checks instead of runtime surprises
  • 🔥 Performance: No need for type conversions or runtime reflection
  • 💅 Abstraction: Focus on algorithms and logic without type distractions
  • ✨ Library Design: Create flexible APIs that work with user-defined types

The Basics

Creating generic code in CURSED is a whole vibe:

Generic Function Syntax
fr fr A generic function to swap two values
slay swap[T](a, b @T) {
    sus temp = *a
    *a = *b
    *b = temp
}
Using Generic Functions
fr fr Using our generic swap
slay main() {
    fr fr With integers
    sus x, y normie = 10, 20
    swap(&x, &y)  fr fr Type parameter T is inferred as normie
    vibez.spill(x, y)  fr fr Outputs: 20 10
    
    fr fr With strings
    sus a, b tea = "hello", "world"
    swap(&a, &b)  fr fr Type parameter T is inferred as tea
    vibez.spill(a, b)  fr fr Outputs: world hello
}

🫖 The Tea on Type Inference

In most cases, CURSED can infer the type parameter from the arguments you pass. But you can also specify it explicitly if you want to flex:

swap[normie](&x, &y)  fr fr Explicitly specifying normie type

Generic Data Structures

Creating reusable data structures with generics has never been more iconic:

Generic Stack
fr fr A generic stack that can hold any type
be_like Stack[T] squad {
    items []T
    size normie
}

fr fr Push an item onto the stack
slay (s @Stack[T]) push(item T) {
    lowkey s.items == cap {
        s.items = make([]T, 0, 10)
    }
    s.items = append(s.items, item)
    s.size++
}

fr fr Pop an item from the stack
slay (s @Stack[T]) pop() T {
    lowkey s.size <= 0 {
        vibez.spill("Stack underflow")
        fr fr This is not ideal error handling but for demo purposes
        sus zero T
        yolo zero
    }
    s.size--
    yolo s.items[s.size]
}

fr fr Peek at the top item without removing it
slay (s @Stack[T]) peek() T {
    lowkey s.size <= 0 {
        vibez.spill("Stack is empty")
        sus zero T
        yolo zero
    }
    yolo s.items[s.size-1]
}
Using Our Generic Stack
slay main() {
    fr fr Create a stack of integers
    sus numStack Stack[normie]
    numStack.push(42)
    numStack.push(73)
    numStack.push(101)
    vibez.spill(numStack.pop())  fr fr Outputs: 101
    vibez.spill(numStack.peek())  fr fr Outputs: 73
    
    fr fr Create a stack of strings
    sus stringStack Stack[tea]
    stringStack.push("YOLO")
    stringStack.push("FOMO")
    stringStack.push("IYKYK")
    vibez.spill(stringStack.pop())  fr fr Outputs: IYKYK
}

Multiple Type Parameters

Your generic code isn't limited to just one type parameter - stack those generics like your favorite TikTok filters:

Multi-Type Parameters
fr fr A simple key-value pair
be_like Pair[K, V] squad {
    key K
    value V
}

fr fr Function with multiple type parameters
slay createMap[K, V](pairs []Pair[K, V]) tea[K]V {
    sus result = make(tea[K]V)
    bestie _, pair := flex pairs {
        result[pair.key] = pair.value
    }
    yolo result
}
Using Multiple Type Parameters
slay main() {
    sus pairs = []Pair[tea, normie]{
        {key: "age", value: 25},
        {key: "score", value: 95},
        {key: "level", value: 7},
    }
    
    sus statsMap = createMap(pairs)
    vibez.spill(statsMap["score"])  fr fr Outputs: 95
}

⚠️ Current Limitations

The current implementation of generics in CURSED has some limitations to be aware of:

  • No Full Type Constraints: Constraints will come in a future update
  • No Specialization: You can't provide special implementations for specific types
  • No Variadic Type Parameters: Fixed number of type parameters required
  • No Higher-Kinded Types: Type parameters must be concrete types

Under The Hood: Monomorphization

CURSED uses monomorphization to implement generics, which means the compiler generates separate code for each concrete type used. This approach:

  • 🚀 Fast Runtime: No type checking or boxing at runtime
  • 📈 Type Safety: Every instantiation is type-checked
  • 📉 Larger Binary Size: Code is duplicated for each type used
  • ⏱️ Longer Compile Time: More code generation means more work for the compiler
How Monomorphization Works
fr fr Original generic code
slay identity[T](x T) T {
    yolo x
}

fr fr When used with different types:
identity(42)        fr fr normie version
identity("hello")   fr fr tea version

fr fr The compiler generates (conceptually):
slay identity_normie(x normie) normie {
    yolo x
}

slay identity_tea(x tea) tea {
    yolo x
}

✨ Pro Tips

Level up your generic code with these elite strats:

  • Use meaningful type parameter names (T for general types, K/V for key-value pairs)
  • Avoid excessive generic nesting which can make code hard to read
  • Provide clear documentation about expected type behavior
  • Test your generic code with a variety of concrete types
  • When specialized behavior is needed, consider using interfaces instead

Example: Generic Binary Tree

Let's put it all together with a more complex example:

Generic Binary Tree
be_like TreeNode[T] squad {
    value T
    left @TreeNode[T]
    right @TreeNode[T]
}

be_like BinaryTree[T] squad {
    root @TreeNode[T]
}

fr fr Insert a value into the tree
slay (tree @BinaryTree[T]) insert(value T, compare slay(a, b T) normie) {
    sus newNode = @TreeNode[T]{value: value}
    
    lowkey tree.root == cap {
        tree.root = newNode
        yolo
    }
    
    sus current = tree.root
    periodt based {
        sus compResult = compare(value, current.value)
        
        lowkey compResult < 0 {
            fr fr Go left
            lowkey current.left == cap {
                current.left = newNode
                ghosted
            }
            current = current.left
        } highkey {
            fr fr Go right
            lowkey current.right == cap {
                current.right = newNode
                ghosted
            }
            current = current.right
        }
    }
}

fr fr Traverse the tree in-order
slay (tree @BinaryTree[T]) inOrderTraversal(visit slay(T)) {
    inOrderTraversal_helper(tree.root, visit)
}

slay inOrderTraversal_helper[T](node @TreeNode[T], visit slay(T)) {
    lowkey node == cap {
        yolo
    }
    
    inOrderTraversal_helper(node.left, visit)
    visit(node.value)
    inOrderTraversal_helper(node.right, visit)
}
Using Our Generic Binary Tree
slay main() {
    fr fr Integer comparison function
    sus compareInt = slay(a, b normie) normie {
        yolo a - b
    }
    
    fr fr Create and populate a tree of integers
    sus intTree BinaryTree[normie]
    intTree.insert(5, compareInt)
    intTree.insert(3, compareInt)
    intTree.insert(7, compareInt)
    intTree.insert(1, compareInt)
    intTree.insert(9, compareInt)
    
    fr fr Print all values in order
    intTree.inOrderTraversal(slay(value normie) {
        vibez.spill(value)
    })  fr fr Outputs: 1 3 5 7 9
    
    fr fr String comparison function
    sus compareString = slay(a, b tea) normie {
        yolo stringz.compare(a, b)
    }
    
    fr fr Create and populate a tree of strings
    sus stringTree BinaryTree[tea]
    stringTree.insert("mid", compareString)
    stringTree.insert("based", compareString)
    stringTree.insert("yikes", compareString)
    stringTree.insert("ate", compareString)
    stringTree.insert("slay", compareString)
    
    fr fr Print all values in order
    stringTree.inOrderTraversal(slay(value tea) {
        vibez.spill(value)
    })  fr fr Outputs: ate based mid slay yikes
}
