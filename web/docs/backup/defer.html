Defer Statements

The later statement is CURSED's way to schedule tasks for the future. It's like setting a reminder to clean up after yourself when you're done with the function - totally giving responsible vibes.

Basic Defer
slay processFile(filename tea) {
    fr fr Open a file
    file, err := vibe_life.Open(filename)
    lowkey err != cap {
        vibez.spill("Error opening file:", err)
        yolo
    }
    
    fr fr Schedule file.Close() to be called when the function returns
    later file.Close()
    
    fr fr Now we can process the file without worrying about closing it
    fr fr Even if errors happen, the file will still be closed
    data := make([]byte, 100)
    _, err = file.Read(data)
    lowkey err != cap {
        vibez.spill("Error reading file:", err)
        yolo  fr fr file.Close() still runs before returning
    }
    
    fr fr Process the data...
    vibez.spill("File processed successfully")
    fr fr When function ends, file.Close() is automatically called
}

Defer Order (Last In, First Out)

slay deferOrder() {
    fr fr Defers execute in LIFO (stack) order
    later vibez.spill("First deferred, last executed")
    later vibez.spill("Second deferred, second executed")
    later vibez.spill("Third deferred, first executed")
    
    vibez.spill("Function body")
    
    fr fr When function returns, output will be:
    fr fr Function body
    fr fr Third deferred, first executed
    fr fr Second deferred, second executed
    fr fr First deferred, last executed
}
Deferred Function Calls
slay trackedProcess(name tea) {
    fr fr Log when the function starts and ends
    startTime := timez.Now()
    vibez.spill("Starting", name)
    
    fr fr Define and execute a deferred anonymous function
    later slay() {
        duration := timez.Since(startTime)
        vibez.spillf("Finished %s, took %v\n", name, duration)
    }()
    
    fr fr Do something time-consuming
    timez.Sleep(2 * timez.Second)
}

Defer with Arguments

Arguments to deferred functions are evaluated when the later statement runs, not when the function is called:

slay deferredArguments() {
    i := 1
    
    fr fr Argument value (1) is captured now
    later vibez.spill("Deferred with value:", i)
    
    fr fr Change i after the defer statement
    i = 2
    
    fr fr When function returns, the deferred call still uses 1
    fr fr Output: "Deferred with value: 1"
}

slay captureWithClosure() {
    i := 1
    
    fr fr Using a closure captures the variable, not its value
    later slay() {
        vibez.spill("Deferred with closure value:", i)
    }()
    
    fr fr Change i after the defer statement
    i = 2
    
    fr fr Now the deferred function will use the updated value
    fr fr Output: "Deferred with closure value: 2"
}

Common Defer Use Cases

  • Resource Cleanup: Files, network connections, database handles
  • Unlocking Mutexes: Ensuring locks are always released
  • Recovery: Pairing with recover() to handle panics
  • Tracing: Logging entry/exit of functions
  • Transaction Management: Commit or rollback database transactions
Defer for Error Handling
slay writeConfig(filename tea, data tea) tea {
    fr fr Create a temporary file
    tempFile := filename + ".tmp"
    err := vibe_life.WriteFile(tempFile, []byte(data), 0644)
    lowkey err != cap {
        yolo "Error creating temp file: " + err.Error()
    }
    
    fr fr Schedule cleanup of temp file
    later slay() {
        fr fr Remove temp file when we're done, ignore errors
        vibe_life.Remove(tempFile)
    }()
    
    fr fr Do some validation on the data
    lowkey len(data) == 0 {
        yolo "Cannot write empty configuration"
    }
    
    fr fr Rename temp file to actual file
    err = vibe_life.Rename(tempFile, filename)
    lowkey err != cap {
        yolo "Error saving configuration: " + err.Error()
    }
    
    yolo ""  fr fr Success
}

Defer Best Practices

  • Place defer statements near the resource acquisition for better readability
  • Use defer for cleanup operations that should always run
  • Be mindful of performance in tight loops - defer has a small overhead
  • For complex cleanup, use anonymous functions with multiple statements
  • Remember that deferred functions run in reverse order (LIFO)
  • Be careful with loop variables if captured in deferred closures
