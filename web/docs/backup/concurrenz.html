Concurrency Sync Thread Safety

The concurrenz package gives you all the tools to keep your concurrent code from going chaotic. It's like setting boundaries in your situationships between goroutines!

✨ Shared State Problems ✨

When multiple goroutines access the same data, you can get race conditions (not the vibe). The concurrenz package provides synchronization primitives to prevent these problems:

Without Synchronization (Bad Vibes)

Two goroutines fighting over the same data = chaos

🚀 Mutex: The Bouncer 🚀

A Mutex (mutual exclusion) is like a club bouncer - only one goroutine gets in at a time:

mutex_example.csd
yeet "concurrenz";

be_like Counter squad {
    value normie;
    lock concurrenz.Mutex;  // Our bouncer
}

slay (c @Counter) Increment() {
    c.lock.Lock();        // Get exclusive access
    later c.lock.Unlock();  // Make sure we release it when done
    
    c.value++;           // Critical section - only one goroutine at a time
}

slay (c @Counter) GetValue() normie {
    c.lock.Lock();
    later c.lock.Unlock();
    
    yolo c.value;
}

��The Tea on Mutexes☕

  • Call Lock() to get exclusive access
  • Call Unlock() when you're done
  • Always use later (defer) with Unlock() to avoid deadlocks
  • Keep the locked section as small as possible for better performance

�� RWMutex: The VIP System ��

When reads are more common than writes, use RWMutex - it allows multiple readers but only one writer:

rwmutex_example.csd
yeet "concurrenz";

be_like SharedConfig squad {
    settings tea[tea]tea;
    lock concurrenz.RWMutex;
}

slay (c @SharedConfig) GetSetting(key tea) tea {
    c.lock.RLock();        // Reader lock - multiple allowed
    later c.lock.RUnlock();
    
    yolo c.settings[key];
}

slay (c @SharedConfig) SetSetting(key tea, value tea) {
    c.lock.Lock();         // Writer lock - exclusive access
    later c.lock.Unlock();
    
    c.settings[key] = value;
}

�� WaitGroup: The After Party ��

Use WaitGroup when you need to wait for multiple goroutines to finish before continuing:

waitgroup_example.csd
yeet "concurrenz";

slay processItems(items []Item) {
    sus wg concurrenz.WaitGroup;
    
    for (sus i = 0; i < len(items); i++) {
        wg.Add(1);  // Add a counter for each goroutine
        
        stan slay(item Item) {
            later wg.Done();  // Signal completion
            
            // Process the item...
            processItem(item);
        }(items[i]);
    }
    
    wg.Wait();  // Block until all goroutines call Done()
    console.print("All items processed!");
}
1
Call Add(n) to set counter
2
Launch goroutines
3
Each calls Done() when finished
4
Main goroutine calls Wait()

🚀 Once: The One-Time Flex 🚀

Need to ensure something happens exactly once, even with multiple goroutines? Once is the vibe:

once_example.csd
yeet "concurrenz";

sus initOnce concurrenz.Once;
sus database @Database;

slay getDatabase() @Database {
    initOnce.Do(slay() {
        // This will execute exactly once, no matter how many
        // goroutines call getDatabase() concurrently
        console.print("Initializing database connection...");
        database = connectToDatabase();
    });
    
    yolo database;
}

�� Cond: The Group Chat ��

A Cond (condition variable) lets goroutines wait until a condition is met:

cond_example.csd
yeet "concurrenz";

be_like Queue squad {
    items []Item;
    lock concurrenz.Mutex;
    notEmpty @concurrenz.Cond;
}

slay NewQueue() @Queue {
    sus q Queue;
    q.notEmpty = concurrenz.NewCond(&q.lock);
    yolo &q;
}

slay (q @Queue) Add(item Item) {
    q.lock.Lock();
    later q.lock.Unlock();
    
    q.items = append(q.items, item);
    q.notEmpty.Signal();  // Notify ONE waiting goroutine
}

slay (q @Queue) Get() Item {
    q.lock.Lock();
    later q.lock.Unlock();
    
    periodt len(q.items) == 0 {
        q.notEmpty.Wait();  // Release lock and wait for signal
        // Lock is reacquired when Wait returns
    }
    
    item := q.items[0];
    q.items = q.items[1:];
    yolo item;
}

Save resources and reduce garbage collection with Pool:

pool_example.csd
yeet "concurrenz";

sus bufferPool = concurrenz.Pool{
    New: slay() collab{} {
        yolo make([]byte, 4096);
    },
};

slay processRequest(data []byte) {
    // Get a buffer from the pool
    sus buffer = bufferPool.Get().([]byte);
    later bufferPool.Put(buffer);  // Return to pool when done
    
    // Use the buffer for processing...
    copy(buffer, data);
    // Do work with buffer...
}

⚠️ No Cap: Common Mistakes ⚠️

Forgetting to Unlock

deadlock_vibes.csd
slay doSomething() {
    mu.Lock();
    // Missing Unlock or defer Unlock!
    
    lowkey someCondition {
        yolo;  // Early return without unlocking = deadlock
    }
    
    // Code continues...
    mu.Unlock();
}

Fix: Always use later mu.Unlock() right after acquiring the lock

be_like Config squad {

slay updateConfig(c Config) {  // Copying by value is sus!

    later c.mu.Unlock();

Quick FAQs ❓

When should I use Mutex vs. Channels?

Use channels when passing ownership of data between goroutines. Use mutexes when multiple goroutines need to read/write shared state without transferring ownership.

What's a deadlock and how do I avoid it?

A deadlock happens when goroutines are waiting for each other in a circular dependency. Avoid by: (1) Always acquiring locks in the same order, (2) Using timeouts, (3) Using later (defer) for unlocking, and (4) Keeping locked sections small.

Do I need synchronization with channels?

Channels provide synchronization on their own for the data being passed through them. However, if you're sharing other data between goroutines outside of channel operations, you might still need mutexes.
