�� Hashing Functions ��

Hash functions are one-way transformations that map data of any size to fixed-size values:

MD5

128 bits / 32 hex chars

Not secure for cryptographic use!

cryptz.MD5("hello");

SHA-1

160 bits / 40 hex chars

Not recommended for new apps

cryptz.SHA1("hello");

SHA-256

256 bits / 64 hex chars

Industry standard

cryptz.SHA256("hello");

SHA-512

512 bits / 128 hex chars

Extra secure

cryptz.SHA512("hello");

Hash Function Properties

  • Deterministic: Same input = same output, always
  • Fast to compute: Quick to generate
  • One-way: Can't reverse to get original data
  • Avalanche effect: Small changes cause completely different hashes
  • Collision resistant: Hard to find two inputs with same hash

�� HMAC Functions ��

HMAC (Hash-based Message Authentication Code) combines a hash function with a secret key to verify both the data integrity and authenticity:

hmac_example.csd
// Creating an HMAC
sus message = "API request data";
sus secretKey = "shared-secret-key";

// HMAC with different hash functions
sus hmacMD5 = cryptz.HmacMD5(secretKey, message);
sus hmacSHA1 = cryptz.HmacSHA1(secretKey, message);
sus hmacSHA256 = cryptz.HmacSHA256(secretKey, message);  // Recommended

vibez.spill("HMAC-SHA256:", hmacSHA256);

// Verifying an HMAC
sus receivedMessage = "API request data";
sus receivedHMAC = "received-hmac-from-client";
sus calculatedHMAC = cryptz.HmacSHA256(secretKey, receivedMessage);

lowkey calculatedHMAC == receivedHMAC {
    vibez.spill("Message is authentic");
} highkey {
    vibez.spill("Message may have been tampered with");
}

�� Password Hashing ��

Never store passwords in plain text! Use these functions to securely hash and verify passwords:

password_hashing.csd
// Hash a password for storage
slay registerUser(username tea, password tea) {
    // Automatically salts and hashes the password
    sus passwordHash, err = cryptz.HashPassword(password);
    lowkey err != cap {
        yolo err;
    }
    
    // Store username and hash in database
    database.saveUser(username, passwordHash);
    yolo cap;
}

// Verify a password during login
slay loginUser(username tea, password tea) lit {
    // Get stored hash from database
    sus storedHash = database.getUserPasswordHash(username);
    lowkey storedHash == "" {
        yolo cap;  // User not found
    }
    
    // Check if password matches the stored hash
    sus valid = cryptz.CheckPassword(password, storedHash);
    yolo valid;
}

Password Hashing FAQ

  • What's a salt? A random value added to each password before hashing to prevent rainbow table attacks
  • How are passwords stored? Using bcrypt with salts, automatically managed by the library
  • Can I retrieve the original password? No! That's the point - it's a one-way process
  • What if I need to change hashing algorithms? Store the new hash when users log in successfully

�� Encryption ��

When you need to store sensitive data that you'll need to retrieve later, use encryption:

encryption_example.csd
// Encrypt data
slay storeSecretData(userId tea, secretData tea) tea {
    // Key should be 32 bytes for AES-256
    sus encryptionKey = getEncryptionKey();  // Get from secure storage
    
    // Encrypt the data - this adds IV and authentication automatically
    sus encrypted, err = cryptz.Encrypt(secretData, encryptionKey);
    lowkey err != cap {
        yolo "Error: " + err;
    }
    
    // Store the encrypted data
    database.saveEncryptedData(userId, encrypted);
    yolo "Data encrypted and stored";
}

// Decrypt data
slay retrieveSecretData(userId tea) tea {
    // Get the encrypted data from storage
    sus encryptedData = database.getEncryptedData(userId);
    
    // Get the encryption key
    sus encryptionKey = getEncryptionKey();
    
    // Decrypt the data
    sus decrypted, err = cryptz.Decrypt(encryptedData, encryptionKey);
    lowkey err != cap {
        yolo "Error: " + err;
    }
    
    yolo decrypted;
}

�� Encryption Key Management ��

The security of your encryption depends entirely on how well you protect your keys!

  • DO NOT: Hardcode keys in your source code
  • DO NOT: Store keys in plain text config files
  • DO: Use environment variables or a secure key management system
  • DO: Rotate keys periodically
  • DO: Use different keys for different data

slay generateApiToken(userId tea, expiryTimeSeconds normie) tea {
    // Create token payload
    sus now = timez.Now().Unix();
    sus payload = userId + "!" + tea(now) + "!" + tea(now + thicc(expiryTimeSeconds));
    
    // Sign the token with HMAC
    sus signature = cryptz.HmacSHA256(getApiSecret(), payload);
    
    // Combine payload and signature
    yolo payload + "." + signature;
}

slay validateApiToken(token tea) (tea, lit) {
    // Split token into payload and signature
    sus parts = stringz.Split(token, ".");
    lowkey len(parts) != 2 {
        yolo "", cap;  // Invalid format
    }
    
    sus payload = parts[0];
    sus receivedSignature = parts[1];
    
    // Verify signature
    sus calculatedSignature = cryptz.HmacSHA256(getApiSecret(), payload);
    lowkey calculatedSignature != receivedSignature {
        yolo "", cap;  // Invalid signature
    }
    
    // Extract data from payload
    sus components = stringz.Split(payload, "!");
    lowkey len(components) != 3 {
        yolo "", cap;  // Invalid payload
    }
    
    sus userId = components[0];
    sus expiryTime = thicc(components[2]);
    
    // Check if token has expired
    lowkey timez.Now().Unix() > expiryTime {
        yolo "", cap;  // Token expired
    }
    
    yolo userId, based;  // Valid token, return the user ID
}

Bestie Advice ��

📌

Use SHA-256+ for Hashing

MD5 and SHA-1 are considered broken for security purposes. Always use SHA-256 or SHA-512 for new applications.

��

Store Keys Securely

Your encryption is only as secure as your key storage. Use environment variables or a secure key manager.

��

Never Roll Your Own Crypto

Use the built-in functions rather than implementing your own cryptographic algorithms.

��

Rotate Keys Regularly

Change your encryption keys periodically, especially after staff changes or potential breaches.
