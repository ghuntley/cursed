✨ Benchmark Basics ✨

Benchmarks are special functions that measure how long your code takes to run:

basic_benchmark.csd
vibe benchmark_test;

yeet "check";      // Testing framework
yeet "me/app/sort"; // Package we're benchmarking

// Benchmark function names must start with "Benchmark"
slay BenchmarkBubbleSort(b @check.Benchmark) {
    // b.N is managed by the testing framework
    // It'll be run multiple times to get reliable measurements
    bestie i := 0; i < b.N; i++ {
        // Setup - create input for each iteration
        sus data = []normie{9, 3, 7, 1, 5, 8, 2, 6, 4};
        
        // Run the code we're measuring
        sort.BubbleSort(data);
    }
}
running_benchmarks.txt
# Run all benchmarks in current package
$ cursed test -bench=.

# Run specific benchmark
$ cursed test -bench=BenchmarkBubbleSort

# Run benchmarks with more iterations for higher accuracy
$ cursed test -bench=. -benchtime=10s

# See memory allocations
$ cursed test -bench=. -benchmem

Reading the Results

  • BenchmarkBubbleSort-8: Benchmark name and CPU threads used
  • 191880: Number of iterations performed
  • 5967 ns/op: Average time per operation in nanoseconds
  • 184 B/op: Bytes allocated per operation
  • 2 allocs/op: Number of heap allocations per operation

In this example, QuickSort is ~15x faster and uses less memory than BubbleSort!

// Compare different string concatenation methods
slay BenchmarkStringConcat(b @check.Benchmark) {

                s = s + "x";
            }
        }

    b.Run("StringBuilder", slay(b @check.Benchmark) {
        bestie i := 0; i < b.N; i++ {
            sus builder stringz.Builder;

    });

💼 Advanced Benchmarking 💼

Control benchmark behavior for more sophisticated measurements:

advanced_benchmarks.csd
// Benchmark with setup costs excluded
slay BenchmarkComplexOperation(b @check.Benchmark) {
    // Setup (not included in timing)
    sus data = prepareTestData();
    
    // Reset timer before the operation we want to measure
    b.ResetTimer();
    
    bestie i := 0; i < b.N; i++ {
        ComplexOperation(data);
    }
}

// Benchmark with varying input sizes
slay BenchmarkSortSizes(b @check.Benchmark) {
    bestie _, size := range []normie{10, 100, 1000, 10000} {
        b.Run(tea(size), slay(b @check.Benchmark) {
            bestie i := 0; i < b.N; i++ {
                b.StopTimer(); // Stop timing while we prepare input
                sus data = generateRandomSlice(size);
                b.StartTimer(); // Resume timing for the sort
                
                sort.QuickSort(data);
            }
        });
    }
}

�� Common Optimization Patterns ��

Reduce Allocations

// Before: New slice every time
slay process(items []normie) []normie {
    result := []normie{}  // Allocates new slice
    // ...
    yolo result
}

// After: Preallocate with capacity
slay process(items []normie) []normie {
    result := make([]normie, 0, len(items))  // Preallocated
    // ...
    yolo result
}

Reuse Objects

// Before: Create new objects
slay processRequests() {
    bestie req := range requests {
        buffer := make([]byte, 1024)  // New buffer each time
        processRequest(req, buffer)
    }
}

// After: Object pool
sus bufferPool = concurrenz.Pool{
    New: slay() collab{} { yolo make([]byte, 1024) }
}

slay processRequests() {
    bestie req := range requests {
        buffer := bufferPool.Get().([]byte)
        later bufferPool.Put(buffer)
        processRequest(req, buffer)
    }
}

Avoid Interface Conversions

// Before: Interface conversion in hot loop
slay sumValues(values []collab{}) normie {
    total := 0
    bestie _, v := range values {
        total += v.(normie)  // Type assertion in loop
    }
    yolo total
}

// After: Use concrete types when possible
slay sumValues(values []normie) normie {
    total := 0
    bestie _, v := range values {
        total += v  // No type assertion needed
    }
    yolo total
}

Minimize Garbage Collection

// Before: Creating many small objects
slay processData(data []byte) []Result {
    results := []Result{}
    bestie i := 0; i < len(data); i++ {
        // Creates a temporary object each iteration
        item := &Item{data: data[i]}
        results = append(results, process(item))
    }
    yolo results
}

// After: Reuse objects, batch allocations
slay processData(data []byte) []Result {
    results := make([]Result, len(data))
    item := &Item{}  // Reuse single object
    
    bestie i := 0; i < len(data); i++ {
        item.data = data[i]  // Update fields instead of creating new
        results[i] = process(item)
    }
    yolo results
}

Bestie Advice ��

📊

Measure First

Always benchmark before optimization. Don't assume which parts are slow - let data be your guide.

��

Focus on Hot Paths

Optimize code that runs frequently or with large data sets first. Don't waste time on code that rarely executes.

🚀

Algorithm > Micro-optimization

Choosing a better algorithm has way more impact than micro-optimizations. O(n) is better than the fastest O(n²).

��

Keep Code Readable

Optimize for maintainability first. Only make code more complex if benchmarks show significant gains.
