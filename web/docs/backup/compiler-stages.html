�� Lexical Analysis Vibe Check ��

The lexer reads your code character by character and turns it into tokens. It's like taking your essay and breaking it into words:

Source Code → Tokens
slay addNums(a normie, b normie) normie {
    yolo a + b;
}

// Becomes tokens like:
// KEYWORD("slay"), IDENTIFIER("addNums"), LEFT_PAREN,
// IDENTIFIER("a"), KEYWORD("normie"), COMMA, ...

Token Types in CURSED

  • Keywords: slay, sus, yolo, lowkey, etc.
  • Identifiers: Variable and function names
  • Literals: Numbers, strings, based, sus, etc.
  • Operators: +, -, *, ==, etc.
  • Punctuation: {, }, (, ), ;, etc.

�� Parsing: Building The AST ��

The parser takes tokens and builds a tree structure that represents your code:

Tokens → AST
// From tokens to a tree structure:

Function {
    name: "addNums",
    params: [Parameter{name: "a", type: "normie"}, 
             Parameter{name: "b", type: "normie"}],
    returnType: "normie",
    body: Block {
        statements: [Return { 
            expression: BinaryOp {
                left: Variable{name: "a"},
                operator: "+",
                right: Variable{name: "b"}
            }
        }]
    }
}

�� Semantic Analysis: Vibe Check ��

This stage makes sure your code actually makes sense and follows the rules:

  • ✅ Variable declarations before use
  • ✅ Type checking (no adding strings to numbers)
  • ✅ Function call parameter matching
  • ✅ Scope validation
  • ✅ Control flow analysis
Type Checking Example
sus x normie = 5;
sus y tea = "hello";
sus z = x + y;  // Type error! Can't add number and string

💻 Intermediate Representation (IR) 💻

The AST gets converted to a lower-level representation that's closer to machine code but still platform-independent:

LLVM IR Example
define i32 @addNums(i32 %a, i32 %b) {
entry:
  %sum = add i32 %a, %b
  ret i32 %sum
}

Why Use LLVM?

We use LLVM as our compiler backend because:

  • It provides powerful optimizations out of the box
  • It supports multiple target platforms
  • It's well-tested and industry-standard
  • It lets us focus on the frontend (the CURSED language itself)

⚡ Optimization: Making Code Go Brrr ⚡

Optimization passes make your code run faster and use less memory:

Constant Folding

Loop Unrolling

Replaces loops with repeated code

Inlining

Replaces function calls with function body

�� Code Generation: The Final Form ��

This stage turns optimized IR into actual machine code for your CPU architecture:

Machine Code (x86-64)
addNums:
    push rbp
    mov rbp, rsp
    mov DWORD PTR [rbp-4], edi
    mov DWORD PTR [rbp-8], esi
    mov edx, DWORD PTR [rbp-4]
    mov eax, DWORD PTR [rbp-8]
    add eax, edx
    pop rbp
    ret
