Garbage Collection in CURSED

CURSED's garbage collector is like that friend who always cleans up after the party without being asked. No cap, it's a whole vibe.

Core Features

  • 🧹 Mark-and-Sweep Algorithm: Identifies and collects unreachable objects like finding who's not in the group chat
  • ⏱️ Incremental Collection: Splits work into smaller chunks to avoid those awkward pauses mid-convo
  • 🔄 Generational Collection: Gives special treatment to new objects because they're more likely to ghosted quickly
  • 🔍 Memory Layout Optimizations: Keeps related objects close together for that bestie energy
  • 📊 Detailed Stats: Tracks memory usage so you can flex on those performance gains

How It Works

The CURSED garbage collector is totally giving "clean girl aesthetic" for your memory:

1

Marking Phase

The GC starts from "root" objects (like global variables and stack frames) and marks all reachable objects as "alive" - basically like checking who's still active in the group chat.

2

Sweeping Phase

Any objects not marked get collected - that's the "you can't sit with us" moment for unused memory. The space is then available for new allocations.

3

Compaction (Optional)

To avoid memory fragmentation (so messy), the GC can rearrange objects to create larger contiguous free spaces. It's like organizing your closet after doing a massive unhaul.

💅 The Tea on Weak References

When you need to reference objects without preventing garbage collection (for caches, observers, etc.), use weak references. They won't keep objects alive if nothing else references them - they're basically the "no strings attached" of reference types.

Using Weak References
yeet "memory/weak"

fr fr Create a weak reference to an object
sus obj = allocate<SomeType>(value)
sus weakRef = weak.create(obj)

fr fr Try to get the object later - returns cap if collected
sus resolvedObj = weak.resolve(weakRef)

When Does Collection Happen?

The GC runs in these situations:

  • 🧠 Memory Pressure: When available memory gets low, like when your phone storage is about to flop
  • ⏰ Idle Time: During application idle periods for that proactive clean-up energy
  • 📞 Manual Trigger: When you explicitly call collect_garbage() - though that's usually not the move
  • 🔄 Threshold Reached: After a certain number of allocations because we're counting those red flags

Performance Tuning

You can customize the GC's behavior for different application needs:

Configuring the Garbage Collector
yeet "memory/gc"

fr fr Set GC configuration
gc.configure({

    incremental: based,             fr fr Enable incremental collection

⚠️ Common Pitfalls

Even with automatic memory management, you can still create memory issues that are giving red flag:

  • Memory Leaks: Keeping references to objects you don't need anymore (main character syndrome)
  • Excessive Object Creation: Creating too many short-lived objects (serial dater energy)
  • Large Object Retention: Holding onto large objects unnecessarily (digital hoarder vibes)
  • Circular References: Creating reference cycles that confuse the GC (it's complicated relationship status)

Implementing Traceable

For custom types to work with the garbage collector, they need to implement the Traceable trait to let the GC know about all references they hold:

Creating Traceable Types
be_like MyCustomObject squad {
    name tea
    child @AnotherObject  fr fr Pointer to another object
    siblings []@AnotherObject  fr fr Array of pointers
}

fr fr Implement the Traceable trait
slay (obj: @MyCustomObject) trace(visitor: @GCVisitor) {
    fr fr Tell the GC about all pointers we have
    visitor.visit(obj.child)
    
    fr fr Trace array elements
    bestie i := 0; i < len(obj.siblings); i++ {
        visitor.visit(obj.siblings[i])
    }
}

✨ Pro Tip

Keep the CURSED garbage collector in check with these elite strats:

  • Use value types for small, short-lived objects to avoid GC overhead
  • Implement object pooling for frequently created/destroyed objects
  • Periodically check memory stats to catch unexpected growth
  • Use the debug_info() function during development to identify memory issues
  • Consider explicitly nulling references to large objects when done with them
